# HLLM Roadmap

> **定位**: 轻量级 LLM 推理框架  
> **核心**: 单卡优化、易用性、快速迭代  
> **不追求**: 多卡并行、超大规模、企业级功能

---

## 设计理念

### 我们要做的
- ✅ 单卡性能极致优化
- ✅ 开发者体验优先
- ✅ 快速原型验证
- ✅ 跨平台支持 (CUDA/MLX/CPU)
- ✅ 易于定制和扩展

### 我们不做的
- ❌ 多卡并行 (Tensor/Pipeline Parallelism)
- ❌ 超大规模模型 (70B+ 量化支持即可)
- ❌ 企业级调度 (复杂 QoS、抢占)
- ❌ 多模态 (Vision/Audio)
- ❌ 复杂的部署工具链

---

## 版本规划

### v0.2.0 - 性能优化 (当前 → 4月底)

**目标**: 单卡性能提升 50%

#### Features
- [ ] **Flash Attention 2** 集成
  - 使用 xFormers 或 FlashAttention 库
  - 预期提升: 30-40% 吞吐量
  - 工作量: 3-5 天

- [ ] **Prefix Caching**
  - 缓存共享前缀的 KV
  - 对多轮对话提升显著
  - 工作量: 5-7 天

- [ ] **CUDA Graph**
  - 捕获静态计算图
  - 减少 CPU 开销
  - 预期提升: 10-20% 低延迟场景
  - 工作量: 3-5 天

- [ ] **GPTQ/AWQ 量化**
  - 支持 4-bit 量化模型
  - 可以跑 7B/13B 模型在 8GB 显存
  - 工作量: 5-7 天

#### 目标性能 (RTX 3080 Ti, Llama-3.2-1B)
- 当前: 64.6 tok/s
- 目标: **90+ tok/s** (+40%)

---

### v0.3.0 - 推理加速 (5月)

**目标**: 降低延迟，提升用户体验 (性能优先于功能)

#### Features
- [ ] **Speculative Decoding**
  - 使用小模型预测 token
  - 预期提升: 2-3x 速度 (draft 模型)
  - 工作量: 7-10 天

- [ ] **Prompt Lookup Decoding**
  - 在 prompt 中查找可复用片段
  - 无 draft 模型，易于部署
  - 工作量: 5-7 天

- [ ] **Chunked Prefill**
  - 长 prompt 分块处理
  - 减少首 token 延迟
  - 工作量: 5-7 天

- [ ] **Logprobs 支持**
  - 返回 token 概率
  - 对调试和分析有用
  - 工作量: 3-5 天

#### 目标延迟
- 当前首 token: 23ms
- 目标首 token: **< 15ms**

---

### v0.4.0 - 量化与压缩 (6月)

**目标**: 支持更大模型在消费级 GPU 上运行

#### Features
- [ ] **GGUF 支持**
  - llama.cpp 格式兼容
  - 广泛的社区模型
  - 工作量: 5-7 天

- [ ] **动态量化**
  - 运行时 KV Cache 量化
  - 减少显存占用
  - 工作量: 7-10 天

- [ ] **AWQ/GPTQ 优化**
  - 更高效的量化推理
  -  fused GEMM kernel
  - 工作量: 5-7 天

#### 目标支持
- 8GB 显存跑 13B 模型
- 12GB 显存跑 30B 模型
- 24GB 显存跑 70B 模型 (量化)

## 竞争对比

### 与 vLLM 的差异
| 维度 | vLLM | HLLM |
|-----|------|------|
| 定位 | 生产级高性能 | 轻量级易用 |
| 多卡 | ✅ 支持 | ❌ 不支持 |
| 70B+ | ✅ 原生支持 | ✅ 量化支持 |
| 易用性 | 复杂 | **简单** |
| 定制性 | 困难 | **容易** |
| Apple Silicon | ❌ | **✅ 原生** |

### 与 llama.cpp 的差异
| 维度 | llama.cpp | HLLM |
|-----|-----------|------|
| 语言 | C++ | **Python** |
| 生态 | 独立 | **PyTorch 生态** |
| 易用性 | 一般 | **简单** |
| 性能 | 极高 | 中等 |
| 量化 | GGUF | **多种格式** |

## 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |

| 性能不达预期 | 低 | 高 | 设定合理目标 (vLLM 70%) |

## 总结

**HLLM 的核心价值**:
1. **轻量**: 单文件可运行，无复杂依赖
2. **易用**: 5 分钟上手，清晰文档
3. **灵活**: Python 生态，易于定制
4. **跨平台**: CUDA/MLX/CPU 统一接口

**不追求**:
- 多卡并行 (留给 vLLM)
- 超大规模 (量化即可)
- 企业级功能 (保持简单)

**目标用户**:
- 研究者 (快速原型)
- 开发者 (本地开发)
- 爱好者 (个人项目)
- Apple Silicon 用户 (MLX 原生)
